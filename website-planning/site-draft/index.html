PostgreSQL 마이그레이션 안전장치

적용한 뒤에 후회하지 말고,적용하기 전에 막는다.

PostgreSQL 스키마 마이그레이션을 위한 안전 게이트.

모든 변경을 적용하기 전에 실제 데이터베이스에 대조해 규칙 기반으로 위험을 검사하고, dry-run으로 검토한 뒤 단일 트랜잭션으로 적용합니다.

Apple Silicon · 약 122 MiB · PostgreSQL 16 · 로컬 LLM(Ollama, 선택) · MIT

01 dry-run 스택

절반만 적용되는 마이그레이션은 없습니다.

각 변경을 실제 DB에 미리 돌려 스택에 쌓고, Apply All이 전체를 단일 트랜잭션으로 감쌉니다. 전부 커밋되거나, 전혀 커밋되지 않습니다.

dry-run 스택
1 ALTER TABLE orders ADD status text dry-run 정상
2 UPDATE orders SET status='new' dry-run 정상
3 CREATE INDEX CONCURRENTLY idx_st Undo
3개 구문 · 1개 트랜잭션으로 묶임Apply All
02 위험 규칙

막는 데 그치지 않고, 더 안전한 길을 보여줍니다.

lock-queue를 유발하는 17개 작업을 적용 전에 잡아냅니다. 위험 판정은 결정적 규칙으로 내려지며 AI 추론에 기대지 않습니다. 그냥 빨간 X가 아니라, 무중단으로 우회하는 golden-path를 함께 제시합니다.

위험 검사 · ADD_CHECK_VALIDATING LOCK QUEUE
NOT VALID 없는 ADD CHECK는 orders에 ACCESS EXCLUSIVE 락을 잡고 전체 행을 스캔합니다.
권장 해결책 · golden-path
ALTER TABLE orders ADD CONSTRAINT chk NOT VALID;
ALTER TABLE orders VALIDATE CONSTRAINT chk;
03 Ollama 기반 로컬 AI

말로 적어도, 모든 처리는 이 기기 안에서 끝납니다.

로컬 Ollama는 자연어를 SQL로 바꾸고 스키마를 근거로 RAG 검색을 하는 편의 레이어입니다. 위험 판정 같은 안전 판단은 맡지 않습니다. 프롬프트도 결과도 기기를 떠나지 않으며, Ollama가 없어도 SQL을 직접 입력하면 안전 코어는 그대로 작동합니다.

NL → SQL · Ollama (로컬)
orders에 status 컬럼을 추가하고 락 없이 인덱스를 만들어줘.
생성된 SQL · 스키마 RAG로 근거 확보
ALTER TABLE orders ADD COLUMN status text;
CREATE INDEX CONCURRENTLY idx_orders_status
  ON orders (status);
추론은 로컬 Ollama에서 실행됩니다. 프롬프트도 결과도 기기를 떠나지 않습니다.

적용한 뒤에 후회하지 말고, 적용하기 전에 막으세요.

무료 오픈소스. 모든 작업이 로컬에서 동작합니다. 운영 DB에 ALTER를 던지는 순간을 위해 만들었습니다.
