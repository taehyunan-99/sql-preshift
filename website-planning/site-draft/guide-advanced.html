기본기를 익혔다면, 더 깊이

여러 변경을 누적해 한 번에 적용하고, 무결성을 진단하고, 생성된 SQL과 롤백 스크립트를 확인하고, 감사 이력에서 개별 롤백을 실행하고, 자연어 입력을 켜고, 런타임에 DB를 교체하는 방법을 다룹니다. 처음이라면 시작하기부터 보세요.

On this page
Section 7 · Advanced changes

여러 변경 누적 · diff 모드 · n홉

누적 dry-run (Apply All) A1

여러 SQL을 Add to preview로 스택에 쌓아 누적 효과를 한 화면에서 검토한 뒤 all-or-nothing으로 일괄 적용합니다.

Split / Unified diff A2

DiffControls 우상단 토글. Split = 직전 1건 before/after, Unified = 누적 전체 after.

n홉 부분집합으로 변경 관련 테이블만 보기 A3

큰 DB에서 변경에 관련된 테이블만 2~3 levels 깊이로 골라 표시합니다. DiffControls의 카운터에 Showing 12 of 150처럼 표시되어 150개 테이블 중 변경과 무관한 노이즈를 숨깁니다.

Section 8 · Diagnostics

무결성 진단과 생성된 SQL/롤백 확인

무결성 진단 5종으로 고아 행 등 점검 A4

DiagnosticsPanel에서 스키마 무결성을 5가지로 진단합니다. Broken refs(끊긴 참조 = 고아 행), Soft-del refs(소프트 삭제 참조), Isolated(고립 테이블), Estimated FK(추정 FK), Rarely used(거의 안 쓰는 객체). 각 항목에서 Locate in ERD로 해당 노드로 점프합니다. 고아 행 점검은 Broken refs에 해당합니다.

SQL 탭 (Up/Down)으로 실행 SQL과 롤백 SQL 확인하기 A5

실행될 SQL과 자동 생성된 롤백 SQL을 확인하는 곳은 DiagnosticsPanel의 SQL 탭입니다. Up이 실제 실행될 SQL, Down이 앱이 자동 생성한 롤백 SQL이며, 둘 다 Monaco 에디터에 읽기 전용으로 표시됩니다. (이 down_script는 메타 DB의 migration_history에 저장됩니다.)

변경 설명 A6

DiagnosticsPanel의 Explanation이 SQL을 한 문장 자연어로 설명합니다. Ollama 폴백 시 토큰 기반으로 동작하므로 모델이 없어도 됩니다.

Section 9 · Audit history

감사 이력과 개별 롤백

History 드로어 A7

AuditDrawer(History)에서 과거 apply/rollback 액션 타임라인을 보고, 항목별로 개별 Rollback을 실행합니다(시작하기의 적용·롤백에서 다룬 두 번째 롤백 경로).

메타 DB가 추적하는 것

audit_log(액션 타임라인 → 이 드로어), migration_history(적용 SQL + 자동 생성 down_script). 전부 로컬 메타 DB(설치형 SQLite).

자동 롤백 한계 재확인

익명 제약은 자동 롤백 불가 → down_script에 주석으로 남습니다.

Section 10 · Natural-language & models

자연어 입력 켜기와 모델 관리

자연어 입력은 로컬 모델이 ready여야 켜집니다. 여기서 모델을 고르고, 다운로드하고, 교체합니다.

모델 선택 B2

온보딩 ModelPicker 또는 TopBar Model에서 큐레이션 카드를 선택+다운로드, 또는 None(Direct SQL only).

Tier태그용량설명
Lightqwen3:4b3.7GBFastest. 짧은 질문/작은 스키마
Balancedgemma4:e2b8.4GB정확도-속도 균형
Qualitygemma4:latest10.8GB복잡 조인/큰 스키마 최고 정확도
None··Direct SQL only
모델 관리 (ModelSettings) A8

모델 교체/다운로드(인앱 SSE 진행률 1/2 chat → 2/2 embed, manifest → verifying → success, 취소 가능)/Advanced 직접 태그 입력/None.

임베딩 모델 고정

bge-m3(1.2GB)는 고정·자동 동반되며 사용자가 변경할 수 없습니다.

NL 활성 조건

모델이 ready여야 자연어 입력이 활성화됩니다. 미선택/Ollama 미감지 시 SQL only로 동작합니다.

RAG 재색인 A11

NL 정확도를 위해 스키마 임베딩(schema_embeddings, numpy 코사인)을 재색인하며, 스키마가 바뀌면 백그라운드에서 자동으로 갱신됩니다.

Section 11 · Runtime DB

런타임 DB 교체·해제와 전체 스키마 보기

DB 교체 / 연결 해제 A9

TopBar DB pill에서 런타임 중 다른 DB로 교체(파이프라인 초기화) 또는 Disconnect.

View full schema A10

idle 입력창 아래 View full schema로 분석 없이 전체 스키마 ERD를 로드합니다. 200+ 테이블이면 성능 경고가 뜨며, n홉 부분집합(A3)을 권합니다.
