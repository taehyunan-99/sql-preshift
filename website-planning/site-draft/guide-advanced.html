자연어를 제출하면 무슨 일이 일어나나

자연어를 제출하면 → 자동 감지(Detected: Natural Language 90%) → 모델이 자연어를 SQL로 변환 → Analyzing → Preview(ERD diff 시각화) → 위험 진단으로 이어집니다. 예를 들어 users 테이블에 nickname 컬럼을 추가하는 변경은 대개 위험이 없거나, NOT NULL로 추가하면 ADD_NOT_NULL_NO_DEFAULT(Warning, 주황)가 붙습니다.

Safety guards

Safety guards run on every input: one SQL statement at a time (no multi-statement), system schemas (pg_catalog / information_schema / pg_toast) are blocked, and syntax is parsed (sqlglot) before anything runs. Natural-language → SQL output is held to the same deterministic checks. The LLM only drafts; the safety verdict is code, not the model.

무결성 진단 5종으로 고아 행 등 점검 A4

DiagnosticsPanel에서 스키마 무결성을 5가지로 진단합니다. Broken refs(끊긴 참조 = 고아 행), Soft-del refs(소프트 삭제 참조), Isolated(고립 테이블), Estimated FK(추정 FK), Rarely used(거의 안 쓰는 객체). 각 항목에서 Locate in ERD로 해당 노드로 점프합니다. 고아 행 점검은 Broken refs에 해당합니다.

SQL 탭 (Up/Down)으로 실행 SQL과 롤백 SQL 확인하기 A5

실행될 SQL과 자동 생성된 롤백 SQL을 확인하는 곳은 DiagnosticsPanel의 SQL 탭입니다. Up이 실제 실행될 SQL, Down이 앱이 자동 생성한 롤백 SQL이며, 둘 다 Monaco 에디터에 읽기 전용으로 표시됩니다. (이 down_script는 메타 DB의 migration_history에 저장됩니다.) 다만 익명 제약(제약명 미지정)이나 원본 타입·DDL 정보가 없는 일부 변경은 실행 가능한 롤백 SQL 대신 주석으로 표시됩니다(자세한 한계는 Apply 섹션 참고).

변경 설명 A6

DiagnosticsPanel의 Explanation이 SQL을 한 문장 자연어로 설명합니다. Ollama 폴백 시 토큰 기반으로 동작하므로 모델이 없어도 됩니다.

Risk

위험 진단 읽기 (Critical / Warning)

위험 판정은 100% 결정적인 규칙 기반입니다. 같은 입력에는 항상 같은 판정이 나오며 재현 가능하고 환각이 없습니다. LLM은 판정에 관여하지 않고 해설만 덧붙입니다. 각 위험은 작업을 무조건 막는 게 아니라 더 안전한 golden-path(안전 대안)를 제시합니다.

Critical vs Warning, 두 단계 위계

  • Critical / Destructive (빨강 좌측 바): 파괴적이거나 비가역적인 변경. Apply 전에 명시적인 확인을 강제합니다(그냥 적용 버튼만 눌러서는 넘어가지 않음). 7개 규칙이 여기 속합니다. 예: DROP_TABLE, DELETE_WITHOUT_WHERE, DROP_COLUMN, UPDATE_WITHOUT_WHERE, TRUNCATE, ADD_FK_VALIDATING, TABLE_REWRITE_FULL.
  • Warning (주황 좌측 바): 정보성. Apply를 차단하지 않습니다. 잠금/재작성 등 주의 사항을 알리되 그대로 진행할 수 있습니다.

두 단계는 라벨 단어(Critical / Warning)로 구분됩니다.

DROP TABLE을 입력하면

DROP TABLE 입력은 CRITICAL 규칙으로 감지되어 위험 모달에 표시됩니다. 모달은 안전 대안으로 expand-contract를 제시합니다. 먼저 코드에서 그 테이블 참조를 모두 제거하고, 다음 릴리스에서 실제 DROP을 수행하는 순서입니다. Critical이므로 Apply 전에 명시 확인을 거쳐야 적용됩니다. 판정 자체는 모델과 무관하게 결정적입니다.

Size-aware 안내

일부 위험은 실제 행 수나 크기를 주입해 보여줍니다. 예를 들어 Rewrites ~12M rows (4 GB).

대표 위험 두 가지로 읽는 법 익히기

Warning · ADD_CHECK_VALIDATING LOCK QUEUE
NOT VALID 없는 CHECK는 대상 테이블에 ACCESS EXCLUSIVE 락과 전체 스캔을 일으킵니다.
golden-path
ALTER TABLE ... ADD CONSTRAINT chk ... NOT VALID;
ALTER TABLE ... VALIDATE CONSTRAINT chk;
Critical · DROP_COLUMN DESTRUCTIVE
ALTER ... DROP COLUMN은 비가역적이라 Apply 전 명시 확인을 강제합니다.
golden-path · expand-contract
앱이 미참조를 확인한 뒤 다음 릴리스에 DROP.

전체 17개 카탈로그는 Reference에 있습니다. 여기서는 "읽는 법"만 다룹니다.

Apply & Rollback

변경 적용하고 되돌리기

미리보기에 만족하면 적용합니다. 적용은 단일 트랜잭션이라, 하나라도 실패하면 전부 롤백됩니다(all-or-nothing).

Step 01
Review then Apply All
Why
미리본 변경을 실제 DB에 씁니다.
Steps
  1. 여러 SQL을 적용하려면 각 변경을 Add to preview로 스택에 누적합니다(누적 dry-run).
  2. Apply 또는 Apply All 버튼을 누릅니다.
  3. Critical 위험이 포함돼 있으면 Apply 전 명시 확인을 강제합니다.
Preview stack · Apply All
1 ALTER TABLE orders ADD status text dry-run ok
2 UPDATE orders SET status='new' WHERE id<100 dry-run ok
3 CREATE INDEX CONCURRENTLY idx_st dry-run ok
3 statements · wrapped in one transactionApply All

단일 트랜잭션 동작: Apply All은 스택의 모든 statement를 하나의 트랜잭션으로 묶어 적용합니다. 하나라도 실패하면 전부 롤백되어 아무것도 적용되지 않습니다. 부분 적용은 발생하지 않습니다.

What you'll see
확인 문구는 다음과 같습니다. All N previewed changes will be applied in a single transaction. If any statement fails, none of them are applied.
Destructive

Applying writes the change to your connected PostgreSQL 16 target DB. This is the real change, not a preview.

전부 로컬이며, 연결된 target DB에만 적용됩니다(메타 DB는 건드리지 않습니다).

Step 03
Roll back (두 가지 경로)
두 경로는 서로 다릅니다
  1. 방금 적용분 전체 롤백, CompletedBar의 Rollback: 방금 Apply/Apply All로 적용한 변경 전체를 역순으로 한 번에 되돌립니다. 적용 직후 "방금 그거 취소" 용도입니다.
  2. 과거 항목별 개별 롤백, AuditDrawer의 History: 감사 이력 타임라인에서 과거의 특정 적용 항목을 골라 개별적으로 Rollback합니다. 방금 것이 아니라 이력 속 임의 항목을 되돌릴 때 씁니다.

두 경로 모두, 앱이 적용 시 자동 생성한 롤백 스크립트(down_script)로 되돌립니다.

자동 롤백 한계

익명 제약(제약명을 지정하지 않은 제약)은 자동 롤백이 불가능해 down_script에 주석으로 표시됩니다. 이 한계는 "위험을 사전 차단하는 것"과는 다릅니다. 사전 차단(위험 모달의 Critical 확인)은 롤백이 아니라 적용 전 단계입니다. 위 두 Rollback은 이미 적용된 변경을 되돌리는 사후 경로입니다.

여기까지가 적용과 롤백의 전부입니다. 위험 규칙 전체는 레퍼런스에서 찾아보세요.
