위험 규칙 · CRITICAL 7 (Apply 전 명시 확인 강제, 100% 결정적)

Rule감지golden-path
DELETE_WITHOUT_WHEREWHERE 없는 DELETE범위 명시, 대량은 배치 루프
UPDATE_WITHOUT_WHEREWHERE 없는 UPDATE범위 명시, 배치 루프
DROP_TABLEDROP TABLEexpand-contract: 코드 참조 제거 후 다음 릴리스에 DROP
DROP_COLUMNALTER ... DROP COLUMN앱이 미참조 확인 후 다음 릴리스에 DROP

위험 규칙 · WARNING 10 (정보성·비차단, 100% 결정적)

Rule위험golden-path
ADD_NOT_NULL_NO_DEFAULT기존 행 오류nullable 추가 → 백필 → CHECK → SET NOT NULL
ADD_COLUMN_VOLATILE_DEFAULT전체 재작성+락기본값 없이 추가 → 배치 백필 → SET DEFAULT
ALTER_COLUMN_TYPE전체 재작성+ACCESS EXCLUSIVE 락expand-contract
SET_NOT_NULL전체 스캔+락CHECK NOT VALID → VALIDATE → SET NOT NULL
RENAME_COLUMN_OR_TABLE앱 코드 즉시 깨짐expand-contract
ADD_PK_OR_UNIQUE인덱스 빌드 중 쓰기 차단CREATE UNIQUE INDEX CONCURRENTLY → ADD CONSTRAINT USING INDEX
ADD_CHECK_VALIDATING전체 스캔+락NOT VALID → VALIDATE CONSTRAINT
CREATE_INDEX_BLOCKING빌드 중 쓰기 차단CREATE INDEX CONCURRENTLY
CONCURRENTLY_IN_TRANSACTIONTX 안 CONCURRENTLY → 런타임 실패autocommit 단독 실행
CASCADE의존객체 조용히 드롭RESTRICT 먼저 → 영향 확인 → CASCADE

판정은 100% 결정적(LLM 무관). LLM은 해설만 덧붙입니다. CONCURRENTLY_IN_TRANSACTION은 CREATE/DROP 두 경로에 등장하며, 고유 rule 이름 기준 warning 10종입니다.

하드 가드 (입력 단계 콜아웃과 이중 배치)

  • 멀티 statement 금지(한 번에 1개 SQL).
  • 시스템 스키마 차단: pg_catalog / information_schema / pg_toast.
  • 구문 오류 차단(sqlglot 파싱).
  • NL→SQL의 LLM 출력도 동일 결정적 검증(LLM은 초안만).

Troubleshooting

  • macOS the developer cannot be verified → 우클릭 Open(Big Sur 11은 2회).
  • Connection refused → PostgreSQL :5432 기동 확인 또는 데모 DB(:5433/:5434).
  • No model → SQL only로 동작.
  • 빈 분석 결과 → 데모 DB 전환.

FAQ

Does my data leave the machine?

아니오, 전부 로컬(접속 정보·추론 모두 기기 밖으로 안 나감).

Do I need Ollama?

아니오, 자연어 입력에만 필요합니다. SQL 직접 입력은 모델 없이 동작합니다.

Can a bad migration half-apply?

아니오, Apply All은 단일 트랜잭션 all-or-nothing입니다.

Is the risk check AI or rules?

100% 결정적 규칙 기반입니다(LLM 무관). LLM은 해설만 덧붙입니다. 그래서 같은 입력엔 같은 판정이 나오고 재현 가능하며 환각이 없습니다.
