위험을 검토된 변경으로 바꾸는 다섯 가지 보장.

스키마 변경이 실제 데이터베이스에 실행되기 전에 통과시키는 게이트. 무엇을 검사하고, 무엇을 막고, 어떻게 더 안전한 경로를 제안하는지 정확히 보여드립니다.

01 누적 dry-run 스택

모든 변경을 dry-run하고, 쌓은 뒤, 하나의 트랜잭션으로 전부 적용합니다.

각 변경은 실제 데이터베이스를 대상으로 dry-run되어 검토 스택에 쌓입니다. 자유롭게 살펴보고, 순서를 바꾸고, Undo할 수 있습니다. 준비가 되면 Apply All이 스택 전체를 단일 트랜잭션으로 감쌉니다.

  • 절반만 적용되는 마이그레이션이 없습니다. 하나의 트랜잭션이므로 전부 커밋되거나 전혀 커밋되지 않습니다.
  • 커밋 전 검토. 각 항목은 실제로 적용되기 전에 자신의 dry-run 결과를 보여줍니다.
  • Undo와 롤백. 스택에서 항목을 꺼내거나, 적용 전체를 롤백할 수 있습니다.

02 17개 위험 규칙 + golden-path

lock-queue 작업을 그냥 막는 데 그치지 않고, 더 안전한 경로를 제시합니다.

SQLPreShift는 프로덕션에서 실제로 문제를 일으키는 지점, 즉 lock queue를 유발하는 작업에 맞춰 조정된 17개 규칙을 갖추고 있습니다. WHERE 없는 DELETE/UPDATE, DROP, 전체 테이블 재작성, 제약 조건 검증이 실행되기 전에 모두 표시됩니다. 판정은 결정적 규칙으로 내려지므로 같은 입력엔 항상 같은 결과가 나오며, LLM은 판정을 바꾸지 않고 해설만 덧붙입니다.

  • lock-queue 인식. 진짜 위험은 구문 자체가 아니라 그 뒤에 쌓이는 큐입니다.
  • golden-path 대안. ADD CONSTRAINT … NOT VALID 후 VALIDATE, 무중단으로요.
  • 벽이 아닙니다. 모달은 빨간 X만 보여주는 게 아니라 위험과 그것을 우회하는 방법을 설명합니다.
위험 검사에 대해 읽기
위험 검사 · ADD_CHECK_VALIDATING LOCK QUEUE
NOT VALID 없는 ADD CHECK는 orders에 ACCESS EXCLUSIVE 락을 잡고 전체 행을 스캔합니다.
스캔이 끝날 때까지 읽기·쓰기를 차단합니다 · 약 1.2M행.
golden-path
ALTER TABLE orders ADD CONSTRAINT chk NOT VALID;
ALTER TABLE orders VALIDATE CONSTRAINT chk;
03 규모 인식 영향도

같은 구문이라도 100행과 1억 행에서는 다르게 읽힙니다.

락 시간이 테이블 크기에 비례하는 위험(전체 재작성, 인덱스 빌드, FK 검증, 타입 변경)에는 대상 테이블의 추정 행 수와 디스크 크기가 함께 표시됩니다. lookup 테이블에 대한 재작성은 괜찮지만, 가장 큰 fact 테이블에 대한 동일한 재작성은 점검 시간(maintenance window)이 됩니다.

  • 추정 행 수와 크기가 규모에 민감한 경고에 인라인으로 표시됩니다.
  • 잡음보다 맥락. 심각도는 구문 유형만이 아니라 실제 영향에 따라 조정됩니다.

04 읽기 전용 무결성 진단

연결하는 순간 다섯 가지 검사가 실행되며, 아무것도 바꾸지 않습니다.

연결 시 SQLPreShift는 깨진 참조 무결성을 포함한 다섯 가지 읽기 전용 진단을 실행합니다. 읽기만 합니다. 아무것도 기록되거나 변경되지 않으며, 곧 손댈 데이터베이스의 형태를 알려주기 위해 존재합니다.

  • 설계상 읽기 전용. 진단은 관찰할 뿐, 절대 기록하지 않습니다.
  • 현장에 맞춘 조정. 제약 조건이 없는 것은 깨진 게 아니며, 정말로 고아가 된 참조만 경고합니다.

05 로컬 LLM (선택)

자연어를 SQL로, 모든 토큰이 사용자의 기기 안에 머뭅니다.

로컬 Ollama 모델이 자연어를 SQL로 바꾸며, 스키마에 대한 RAG로 근거를 확보합니다. 위험 판정 같은 안전 판단은 규칙 코어가 맡고, LLM은 초안 생성과 해설만 담당합니다. DB 접속 정보와 추론은 절대 기기를 떠나지 않습니다. Ollama가 설치되어 있지 않다면? SQLPreShift는 SQL을 직접 입력하도록 안내합니다. 안전 코어는 LLM 없이도 작동합니다.

  • 100% 로컬 추론. Apple Silicon GPU에서 Ollama로 실행됩니다.
  • RAG 임베딩이 실제 스키마에 생성을 고정합니다 (1024-dim).
  • 완전히 선택 사항. 핵심 기능은 LLM 없이도 전부 작동합니다.

ALTER TABLE orders ADD COLUMN status text;
CREATE INDEX CONCURRENTLY idx_orders_status

추론은 로컬 Ollama 모델에서 실행됩니다. DB 접속 정보와 프롬프트는 절대 기기를 떠나지 않습니다.
