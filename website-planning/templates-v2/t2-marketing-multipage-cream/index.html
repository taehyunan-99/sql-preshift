What it does

Five ways SQLPreShift catches the migration you'd regret.

Each feature works on your live database, before anything changes. The natural-language layer is optional — the safety core runs without an LLM.

01

Cumulative dry-run stack + single-transaction apply

Each change is dry-run against the real DB, then pushed onto a stack you review and Undo. Apply All wraps the whole stack in one transaction — there is no half-applied migration.

dry-run · undo · apply all · rollbackHow it works
02

18 risk rules + golden-path alternatives

Detects lock-queue operations — WHERE-less DELETE/UPDATE, DROP, full table rewrites, constraint validation. It doesn't just block: it proposes a zero-downtime path (ADD CONSTRAINT … NOT VALID, then VALIDATE).

lock-queue · zero-downtimeHow it works
03

Size-aware impact

Every risk warning carries the target table's estimated row count and size. 100 rows and 100M rows read very differently.

04

Read-only integrity diagnostics on connect

On connection, four read-only checks run — including broken referential integrity. Nothing is changed.

Local LLM for NL→SQL + RAG (optional)

A local Ollama turns natural language into SQL. Credentials and inference stay local. No Ollama? Enter SQL directly — the core still works.

Why trust it

No social proof here — just verifiable safety guarantees.

Single-transaction apply

Apply All commits the whole reviewed stack inside one transaction. If anything fails, the entire batch rolls back — never a half-applied migration.

All local, nothing persisted

Stateless and one-shot. Credentials and inference stay on your machine and are never written to disk. The metadata store is a single local SQLite file.

Read-only on connect

The four integrity diagnostics that run on connection only SELECT. They observe; they never write. Diagnostics change nothing.
