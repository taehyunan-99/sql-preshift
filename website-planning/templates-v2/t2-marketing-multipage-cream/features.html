Features

Five guarantees that turn a risky migration into a reviewed one.

SQLPreShift isn't a daily editor. It's the gate you pass a schema change through before it ever runs against a real database. Here is exactly what it checks.

01 Cumulative dry-run stack

Dry-run every change, stack it, then apply all in one transaction.

Each change is dry-run against your real database and pushed onto a review stack. You inspect, reorder, and Undo freely. When you're ready, Apply All wraps the entire stack in a single transaction.

  • No half-applied migrations. One transaction means it all commits or none does.
  • Review before commit. Each entry shows its dry-run result before anything is real.
  • Undo & Rollback. Pop items off the stack, or roll back the whole apply.

02 18 risk rules + golden path

It doesn't just block lock-queue operations — it offers a safer route.

SQLPreShift carries 18 rules tuned for what actually hurts in production: the lock queue. WHERE-less DELETE/UPDATE, DROP, full table rewrites, and constraint validation are all flagged before they run.

  • Lock-queue aware. The real danger isn't the statement — it's the queue behind it.
  • Golden-path alternatives. ADD CONSTRAINT … NOT VALID then VALIDATE — zero downtime.
  • Not a wall. The modal explains the risk and the way around it, not just a red X.
Read about Risk Check
Risk Check — Rule 12 LOCK QUEUE
VALIDATE CONSTRAINT takes an ACCESS EXCLUSIVE lock on orders.
Blocks reads & writes while the queue drains · est. 1.2M rows.
Golden path
ALTER TABLE orders ADD CONSTRAINT chk NOT VALID;
ALTER TABLE orders VALIDATE CONSTRAINT chk;
03 Size-aware impact

The same statement reads differently on 100 rows and 100 million.

Every risk warning is paired with the target table's estimated row count and on-disk size. A rewrite on a lookup table is fine; the same rewrite on your largest fact table is a maintenance window.

  • Estimated rows & size shown inline with each warning.
  • Context over noise. Severity scales with real impact, not statement type alone.

04 Read-only integrity diagnostics

The moment you connect, four checks run — and change nothing.

On connection, SQLPreShift runs four read-only diagnostics, including broken referential integrity. They only read. Nothing is written, nothing is altered — they exist to tell you the shape of the database you're about to touch.

  • Read-only by design. Diagnostics observe; they never write.
  • Field-tuned. No-constraint isn't broken — only genuinely orphaned references warn.

05 Local LLM (optional)

Natural language to SQL — every token stays on your machine.

A local Ollama model turns plain English into SQL, with RAG over your schema for grounding. Credentials and inference never leave your machine. No Ollama installed? SQLPreShift guides you to type SQL directly — the safety core works without any LLM.

  • 100% local inference via Ollama on your Apple Silicon GPU.
  • RAG embeddings ground generation in your real schema (1024-dim).
  • Fully optional. The core features run with no LLM at all.
