Core capabilities

Five things that stand between you and a bad migration.

Each one runs against your real database. Nothing here is decorative — every screen below is the actual product surface.

01 / DRY-RUN

Cumulative dry-run stack + single-transaction apply

Each change is dry-run against the real DB, then pushed onto a stack. Review with Undo, reconsider, reorder — then Apply All wraps the whole stack in one transaction. There is no in-between, half-applied state.

  • Stage changes incrementally, validated one by one
  • Undo any layer before committing
  • All-or-nothing: one BEGIN to COMMIT for the full stack
Dry-run stack3 staged
1CREATE INDEX CONCURRENTLY idx_orders_statuspassed
2ALTER TABLE customers ADD COLUMN tier textpassed
3UPDATE orders SET status = 'closed'review
1 needs review · 0 errors Apply All — single TX
02 / RISK RULES

18 risk rules + golden-path alternatives

SQLPreShift detects operations that trigger a lock queue — WHERE-less DELETE/UPDATE, DROP, full table rewrites, constraint validation, and more. It doesn't just block you: it offers the zero-downtime alternative.

The classic example: turn a blocking ADD CONSTRAINT into ADD CONSTRAINT … NOT VALID followed by VALIDATE CONSTRAINT.

Risk Check1 of 18 rules fired
Warningblocking_constraint_validation

Validating this foreign key takes an ACCESS EXCLUSIVE lock and scans the whole table — every read and write on orders queues behind it.

  NOT VALID;

ALTER TABLE orders
  VALIDATE CONSTRAINT fk_cust;
03 / IMPACT

Size-aware impact

Every risk warning carries the estimated row count and size of the target table. The same SET NOT NULL reads very differently against 100 rows versus 100 million — and SQLPreShift makes that scale visible before you decide.

  • Estimated rows and on-disk size beside each warning
  • Severity scaled to the actual table, not the statement

04 / DIAGNOSTICS

Read-only integrity diagnostics on connect

The moment you connect, SQLPreShift runs four read-only diagnostics — broken referential integrity among them. It inspects, it reports, and it changes absolutely nothing.

  • Runs automatically the instant a connection opens
  • Strictly read-only — no writes, no side effects

Download and open the disk image

Grab SQLPreShift-*.dmg from the Releases page, then double-click it to mount. A Finder window opens showing the app and an Applications shortcut.

Drag the app into Applications

Drag SQLPreShift.app onto the Applications folder in that same window. That's the whole install — you can eject the disk image afterward.

Unlock the first launch

The app isn't notarized yet, so a plain double-click is blocked by macOS Gatekeeper. Clear it once, either way below — afterward it opens normally.

! Option A — Right-click to openIn Finder, right-click SQLPreShift.app and choose Open. In the warning dialog that appears, click Open again. macOS remembers your choice.

Usage guide

Walking the full flow, one screen at a time.

What each step is, and exactly what you do on screen — from first connection to a clean rollback.

Step 01

Connect

Open the app and enter your PostgreSQL connection in the top bar — host, port, database, user. Credentials are held in memory for the runtime only; nothing is persisted to disk.

The instant the connection succeeds, the four read-only integrity diagnostics run and report on the ERD. Nothing is modified — this is a pure inspection.

+Integrity diagnostics4 checks · read-only
!Broken referential integrity2 tables flagged
Step 02

Input

Type SQL directly into the central input, or describe the change in plain English and let the local model draft the SQL for you. You can edit the generated statement before going further.

If Ollama isn't installed, the natural-language box is replaced with a prompt to enter SQL directly — the rest of the flow is identical.

InputNLSQL
drop the legacy "fax" column from supplierslocal only
ALTER TABLE suppliers DROP COLUMN fax;
Step 03

ERD Diff

The change is rendered as a schema diff on the ERD. Toggle between Split (before and after side by side) and Unified (one diagram, additions and removals color-coded) using the toggle in the panel header.

Added columns read green, removed ones read red and struck through — so the shape of the change is obvious before any risk analysis.

Step 04

Risk Check

Before staging, the 18 risk rules evaluate the statement. If one fires, a modal explains the impact in plain terms, attaches the size-aware row count and table size, and — where one exists — shows the golden-path alternative as ready-to-use SQL.

You decide: accept the safe rewrite, keep your original, or cancel. Nothing is staged until you choose.

Risk Checkwarning
Warningwhereless_update

This UPDATE has no WHERE clause — it rewrites every row and holds a lock the whole time.

table ordersest. rows 41.8M
Step 05

Dry-run Stack

Accepted changes are dry-run against the real database and pushed onto the cumulative stack. Each entry shows its dry-run result — passed, or flagged for review. Keep adding, and use Undo to peel back any layer you reconsider.

The stack is your full migration plan, assembled and validated piece by piece before a single byte is committed.

Dry-run stack3 staged
1ALTER TABLE suppliers DROP COLUMN faxpassed
2ALTER TABLE customers ADD COLUMN tier textpassed
3CREATE INDEX idx_orders_statusreview
Step 06

Apply All

When the stack reads the way you want, hit Apply All. SQLPreShift wraps the entire stack in a single transaction — one BEGIN, every change, one COMMIT. If any statement fails, the whole batch rolls back; you never land in a half-applied state.

Apply · single transactioncommitted
tx>BEGIN;
applied 3 / 3 changes
tx>COMMIT;
[done] stack applied atomically · 412 ms
Step 07

Rollback

Something looks off after applying? Roll the migration back cleanly. Because the apply was a single transaction, undoing it is just as atomic — the schema returns to exactly where it started, with no orphaned partial changes left behind.

Rollbackreverted
tx>ROLLBACK;
[reverted] schema restored to pre-apply state · clean
