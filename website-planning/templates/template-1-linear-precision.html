Capabilities

Five layers between a risky migration and your data.

Not a daily database client — a gate that only matters at the moment a change could hurt.

01

Cumulative dry-run stack with single-transaction apply

Each change is dry-run against your real database, then pushed onto a stack you can review and Undo. When you're ready, Apply All wraps the whole stack in one transaction — there is no intermediate, half-applied state to clean up.

  • Every statement validated against the live schema before it stacks
  • Undo any step; reorder your review without re-running the world
  • Apply All commits or rolls back as a single atomic unit

02

18 risk rules with golden-path alternatives

SQLPreShift detects operations that trigger lock queues — DELETE/UPDATE without a WHERE, DROP, full table rewrites, constraint validation, and more. It doesn't just block you: it proposes the no-downtime path.

  • Catches the operations that actually take production down
  • Rewrites the risky move into a safe sequence
  • ADD CONSTRAINT … NOT VALID, then VALIDATE — never a blocking scan
Risk check
Lock queuerule R-07
ADD FOREIGN KEY scans & locks the whole table
Validating a new constraint takes an ACCESS EXCLUSIVE lock while it reads every row — writes queue behind it.
Golden path
ADD CONSTRAINT fk_customer … NOT VALID;
VALIDATE CONSTRAINT fk_customer;
03

Size-aware impact

Every risk warning carries the target table's estimated row count and size. The same SET NOT NULL reads very differently at 100 rows versus 100 million — so the warning tells you which one you're looking at.

  • Estimated rows and on-disk size attached to each warning
  • Severity scales with the table you'd actually touch
  • Turns an abstract rule into a concrete blast radius
Impact estimate

04

Read-only integrity diagnostics on connect

The moment you connect, SQLPreShift runs four read-only diagnostics — including broken referential integrity — to show the shape of the database before you change anything. It reads. It never writes.

  • Runs automatically at connection time, no setup
  • Tuned against real-world false positives, not naive checks
  • Strictly read-only — your data is untouched

Prefer source? Run the full stack with demo databases: build from source.

Get the .dmg from Releases

Open the Releases page and download SQLPreShift-*.dmg (~122 MiB).

Drag the app into Applications

Double-click the downloaded .dmg to mount it, then drag SQLPreShift.app into your Applications folder. Eject the disk image when you're done.

Unlock the first launch

The build isn't notarized yet, so a normal double-click is blocked by macOS Gatekeeper. Do one of the two below — just once. Every launch afterward works normally.

Option A — Right-click > Open. In Applications, right-click (or Control-click) SQLPreShift.app, choose Open, then click Open again in the warning dialog. This adds a one-time exception.

Option B — remove the quarantine attribute from a terminal:

Terminal 
$ xattr -dr com.apple.quarantine /Applications/SQLPreShift.app

Optional — enable natural-language input

Natural language to SQL needs a local Ollama. Skip this and you can still type SQL directly — every safety feature works without it. To turn it on, pull the two models:

Terminal · Ollama models 
$ ollama pull gemma4:latest   # NL to SQL + explanations
$ ollama pull bge-m3:latest   # RAG embeddings, 1024-dim
Ollama runs on the host (Metal GPU), not inside Docker. Start it before launching the app so SQLPreShift can reach it.

Alternative — run from source

Want the demo databases? Clone the repo and bring up the full stack. This boots Postgres with the 92-table ERP and the Pagila sample DB alongside the backend and frontend. The metadata store is a single SQLite file.

Terminal · from source 
$ cp .env.example .env
$ docker compose up -d   # ERP + Pagila demo DBs, e.g. localhost:5433
Usage guide

Walk the gate, step by step.

Each step is one screen. You stay in control the whole way — nothing commits until Apply All.

1

Connect Connect

Enter the host, port, database, and credentials for your PostgreSQL 16 target and connect. Credentials live in memory for the session only — SQLPreShift is stateless and one-shot, so nothing is persisted to disk. The moment the connection succeeds, the four read-only integrity diagnostics run automatically.

Top bar shows a live connection dot and the active database name
2

Input Input

Describe the change in plain English and let the local model draft SQL, or paste SQL directly into the editor. If Ollama isn't running, the natural-language box steps aside and you work in raw SQL — the gate behaves identically either way.

A centered input panel; toggle between natural language and SQL
3

ERD Diff ERD Diff

The change renders as a schema diff on the ERD canvas, scoped to the affected tables and their n-hop neighbors. Toggle Split to see before and after side by side, or Unified to see additions and removals highlighted on one graph. Opt in to the full schema when you need the wider context.

Added columns and tables glow in the success tone; removals in error
4

Risk Check Risk

Before anything stacks, the 18 risk rules evaluate the statement. If it could trigger a lock queue or rewrite, a modal explains the risk, attaches the target table's estimated rows and size, and offers the golden-path alternative you can adopt with a click — for example, splitting a constraint into NOT VALID then VALIDATE.

A warning modal with impact figures and a safe-alternative button
5

Dry-run Stack Dry-run

Once you accept a statement, it's dry-run against the real database and pushed onto the cumulative stack. Keep adding changes — each one validates on top of the ones before it, so the stack always reflects the real end state. Use Undo to pop the last step and review the whole plan before committing.

A right-hand stack panel; each item shows its dry-run result
6

Apply All Apply All

When the plan looks right, Apply All wraps the entire stack in a single transaction and commits. Either every change lands together or none of them do — there is never a partially applied schema to untangle afterward.

One primary button at the foot of the stack: Apply All · single transaction
7

Rollback Rollback

If a committed change needs reversing, Rollback walks it back cleanly. Combined with the single-transaction apply, you always have a defined way back — the gate is there to make sure the way back is never a guess.

A reverse action restores the prior schema state
