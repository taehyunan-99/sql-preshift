The case for a gate

A migration is the one command you can't take back.

Schema changes don't fail loudly. They fail quietly and late — a SET NOT NULL that holds an ACCESS EXCLUSIVE lock while a hundred million rows are scanned, a DELETE without a WHERE, a column rewrite that queues every read behind it. By the time the alert fires, the table is already locked and the rollback is a postmortem.

Most tooling helps you write the migration faster. Almost none helps you not run the wrong one. SQLPreShift sits exactly there: between intent and execution, where the cost of being wrong is highest and the chance to stop it is lowest.

"Don't regret after applying — stop it before."

It is not a daily driver. It is a checkpoint for the moment that matters — dry-running each change against your real schema, surfacing the lock-queue risks with the actual table size behind them, and offering the no-downtime path instead of merely blocking you.

What it does

Five capabilities, one purpose: catch it before it runs.

No fabricated stats, no enterprise theater. Just the things a gate has to do well.

01

Cumulative dry-run stack with single-transaction apply

Every change is dry-run against the live database first, then pushed onto a stack you can review. Undo any step, reorder your thinking, and when you're ready, Apply All wraps the whole stack in one transaction — so there is no intermediate half-applied state. It commits as a unit or not at all.

02

18 risk rules with golden-path alternatives

SQLPreShift detects the operations that trigger lock queues — DELETE / UPDATE without a WHERE, DROP, full-table rewrites, constraint validation, and more. It doesn't just block you: it proposes the no-downtime alternative, like turning a validating constraint into ADD CONSTRAINT… NOT VALID followed by a separate VALIDATE.

Block, then offer the safe path — not a dead end.

Risk check
LOCK SET NOT NULL rewrites the whole table under ACCESS EXCLUSIVE.customers · full scan blocks reads & writes
Golden path ADD CONSTRAINT ck NOT VALID;  then  VALIDATE CONSTRAINT ck;
03

Size-aware impact

A risk warning means little without scale. SQLPreShift attaches the target table's estimated row count and size to every warning — because the same SET NOT NULL reads very differently against 100 rows than against 100 million.

Prerequisites

  • macOS on Apple Silicon — the only supported platform.
  • A reachable PostgreSQL 16 database to connect to (the source build ships demo databases).
  • Ollama — optional, only for natural-language input.

Download the disk image

Grab SQLPreShift-*.dmg from the GitHub Releases page (about 122 MiB).

Drag the app into Applications

Open the .dmg and drag SQLPreShift.app onto your Applications folder, like any other Mac app.

Unlock the first launch

The build isn't notarized yet, so a plain double-click is blocked by macOS Gatekeeper. You only have to do this once — afterwards it opens normally.

Caution Don't drag it to the Trash if you see "cannot be opened". That dialog is expected for an unsigned app — use one of the two methods below to allow it.

Option A — Right-click to Open. In Applications, right-click SQLPreShift.app and choose Open, then click Open again in the warning dialog.

Option B — Remove the quarantine flag in Terminal:

Terminal 
$ xattr -dr com.apple.quarantine /Applications/SQLPreShift.app

Enable natural language (optional)

SQLPreShift's core features all work without an LLM. To use natural-language input, install Ollama and pull the two local models — one for NL→SQL and one for RAG embeddings.

Tip Skip this entirely if you'd rather type SQL directly — the app will simply guide you to the SQL input.
Terminal · NL to SQL 
$ ollama pull gemma4:latest    # natural language to SQL + explanations
Terminal · RAG embeddings 
$ ollama pull bge-m3:latest    # RAG embeddings, 1024-dim
Note Ollama runs on the host (Metal GPU), not in a container. Start it before launching SQLPreShift so the model is reachable.

Prefer to run from source?

Clone the repo, copy the example env, and bring the stack up. This boots the demo databases too — a 92-table ERP schema and Pagila.

Terminal · from source 
$ cp .env.example .env # seed local config
$ docker compose up -d  # postgres + backend + frontend + demo DBs
Note Demo databases come up on local ports (e.g. localhost:5433). The metadata store is a single SQLite file — credentials are never persisted, they live only for the runtime session.

04 Risk Check

Read the impact, take the safe path

A modal opens with the matched risk rules, the estimated impact sized against the real table, and — when one exists — the no-downtime alternative you can adopt with one action.

  • Each warning shows the rule, the target table, and its row count and size.
  • Where a golden path exists, apply it instead of the risky statement.
  • Acknowledge the risk to continue, or revise the input.
Risk check1 warning
REWRITE UPDATE on every row rewrites the table.orders · ~4.2 M rows · 1.1 GB
Golden path Backfill in batches, then SET NOT NULL with a validated constraint.

05 Dry-run Stack

Accumulate and review before committing

Each verified change drops onto the dry-run stack. Keep building, inspect the order, and Undo anything you reconsider — nothing has touched the database yet.

  • Stacked items show their dry-run result: passed or needs review.
  • Undo pops the last change off the stack cleanly.
  • The stack is your full plan, assembled before a single commit.

06 Apply All

Commit the whole plan as one transaction

Apply All wraps every staged change in a single transaction. It all commits, or nothing does — the database is never left in a half-migrated state.

  • One action runs the entire reviewed stack.
  • A failure anywhere rolls the whole transaction back automatically.
  • Success is atomic — the schema moves in one consistent step.

07 Rollback

Reverse cleanly if it reads wrong

If something looks off after applying, roll the migration back. The history of what you applied stays visible, so reversing is a deliberate, reviewable action — not a scramble.

  • Review the applied changes in order.
  • Roll back to return the schema to its prior state.
  • Re-enter the workflow with a corrected plan.
