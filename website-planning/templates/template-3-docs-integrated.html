What it does

Five capabilities, one safety gate

SQLPreShift isn't a daily driver. It's the gate that stands between a risky migration and your production schema — and only opens when it's safe.

FEATURE 01

Cumulative dry-run stack + single-transaction apply

Each change is dry-run against the real database, then pushed onto a stack. Review with Undo, and Apply All wraps the whole stack in a single transaction. No half-applied in-between state — ever.

01ALTER TABLE orders ADD COLUMN statusdry-run ok
02CREATE TABLE shipments (...)dry-run ok
03CREATE INDEX CONCURRENTLY ...staged
FEATURE 02

18 risk rules + golden-path alternatives

Detects operations that trigger lock queues — DELETE/UPDATE without WHERE, DROP, full table rewrites, constraint validation and more. It doesn't just block: it proposes a zero-downtime path.

Lock queue risk on a hot table
ALTER TABLE orders ADD CONSTRAINT fk_cust ...;
# rewrites + validates under ACCESS EXCLUSIVE
ADD CONSTRAINT ... NOT VALID; VALIDATE CONSTRAINT ...;
FEATURE 03

Size-aware impact

Every warning carries the target table's estimated row count and size. The same SET NOT NULL reads very differently at 100 rows versus 100 million.

est. rows · on-disk size
FEATURE 04

Read-only integrity diagnostics on connect

The moment you connect, SQLPreShift runs 4 read-only diagnostics — broken referential integrity among them. It inspects only; it changes nothing.

read-only · zero writes
FEATURE 05

Local LLM for NL to SQL + RAG

A local Ollama turns plain English into SQL. Credentials and inference stay on your machine — nothing leaves for the cloud. No Ollama? Core features still work; just enter SQL directly.

optional · runs offline

Requirements

SQLPreShift is a native macOS app for Apple Silicon. The natural-language feature is optional and only needs a local Ollama — everything else works without it.

  • macOS on Apple Silicon — the .dmg is ~122 MiB.
    required
  • A PostgreSQL 16 target — the database you want to gate. The bundled demo DBs (ERP, Pagila) ship with the source build.
    required
  • Ollama with two models — only for natural-language input and RAG. Skip it and type SQL directly.
    optional

Download & install

One file, three steps. The first launch needs a one-time unlock because the build isn't notarized yet — covered in the next section.

SQLPreShift for macOS

Universal .dmg, signed but not yet notarized. Grab it from GitHub Releases.

macOS|Apple Silicon|SQLPreShift-*.dmg|~122 MiB
Download .dmg or build from source

  1. Download the .dmg

    From the GitHub Releases page, download SQLPreShift-*.dmg.

  2. Open the disk image

    Double-click the downloaded .dmg to mount it. A Finder window opens showing the app and an Applications shortcut.

  3. Drag the app into Applications

    Drag SQLPreShift.app onto the Applications folder in that window. Then eject the disk image.

First launch (unsigned build)

The build isn't notarized yet, so a plain double-click is blocked by macOS Gatekeeper. You only need one of the two unlocks below — once, the first time.

macOS will refuse the first double-click.

You'll see a warning that the app "cannot be opened because the developer cannot be verified." That's expected for an unnotarized portfolio build — not malware. Use either method below to approve it once.

Option A — Right-click and Open

  1. Right-click the app

    In /Applications, right-click (or Control-click) SQLPreShift.app and choose Open.

  2. Confirm in the dialog

    A warning dialog appears — this time it has an Open button. Click it. macOS remembers your choice, so every launch after this is a normal double-click.

Option B — Remove the quarantine flag

Prefer the terminal? Strip the quarantine attribute in one command, then launch normally.

Terminal 
$ xattr -dr com.apple.quarantine /Applications/SQLPreShift.app
This is a one-time unlock.

After either option, SQLPreShift launches like any other app. You won't need to repeat this for future runs (only after re-downloading a new version).

Optional: local LLM for natural language

Want to type plain English instead of SQL? Install Ollama and pull two models. Everything stays on your machine — no cloud calls.

Pull the two models

One model handles NL to SQL and explanations; the other produces the embeddings for RAG.

Terminal · NL to SQL + explanations 
$ ollama pull gemma4:latest
Terminal · RAG embeddings (1024-dim) 
$ ollama pull bge-m3:latest
No Ollama? You're not blocked.

The core safety gate — diff, risk rules, dry-run, apply, rollback — runs entirely without an LLM. If Ollama isn't detected, SQLPreShift simply prompts you to enter SQL directly.

Run from source

Not on Apple Silicon, or want the demo databases? Run the whole stack with Docker Compose — it brings up Postgres plus the ERP (92 tables) and Pagila demo databases.

Terminal · project root 
$ cp .env.example .env && docker compose up -d
What comes up.

Postgres with the ERP and Pagila demo databases (on localhost:5433 and friends). The metadata store is a single SQLite file — credentials are never persisted; they live only at runtime.

1 · Connect

Point SQLPreShift at the PostgreSQL 16 database you want to protect. Credentials are stateless — held only for the session, never written to disk.

Enter the host, port, database, and role. The instant the connection succeeds, the read-only integrity diagnostics run automatically — broken referential integrity and three other checks — surfaced right on the ERD without writing anything.

2 · Input

Describe the change in plain English, or write SQL directly. Both paths feed the same gate.

With a local Ollama running, type a request like "add a status column to orders" and SQLPreShift drafts the SQL for you to review. Without Ollama, switch the toggle to SQL and paste your statement. Either way, the generated or pasted SQL is what gets diffed and risk-checked next — you stay in control of the exact statement.

3 · ERD Diff

See exactly what the statement would change to your schema, visualized as a before/after ERD.

Toggle between Split (current and proposed side by side) and Unified (one diagram with additions and removals marked inline). Added columns and tables are tinted green; removed ones are struck through in red. This is your last visual sanity check before the risk engine weighs in.

4 · Risk Check

The 18-rule engine inspects the statement, estimates its blast radius, and — when it's risky — hands you a zero-downtime alternative.

If a rule fires, a modal explains which rule, the size-aware impact (estimated rows and on-disk size of the target), and a golden-path rewrite you can stage instead. You decide: proceed as-is, or take the safe path.

5 · Dry-run Stack

Every accepted change is dry-run against the real database, then stacked. Build up a whole batch and review it as a unit.

Each entry shows its status — dry-run ok or a flagged warning. Reorder, inspect, or Undo the top of the stack. Nothing is committed yet; this is your staging area before the single-transaction apply.

6 · Apply All

Commit the entire stack inside a single transaction. It's all-or-nothing — there's no half-applied state to clean up.

When you hit Apply All, SQLPreShift wraps every staged statement in one transaction. If any statement fails, the whole batch rolls back automatically and your schema is untouched. If they all succeed, they commit together.

Single transaction = no in-between state.

This is the core guarantee: you never end up with three of five migrations applied. Either the batch lands completely, or not at all.

7 · Rollback

Changed your mind after a commit? Roll the applied batch back.

If a committed change turns out wrong, use Rollback to reverse it. Combined with the integrity diagnostics that re-run on demand, you can confirm the schema is back to a known-good state before moving on. That's the full loop — connect, propose, gate, apply, and undo if needed.

Rollback reverses schema changes, not lost data.

A destructive statement that the risk engine flagged — say a dropped column — can be structurally reversed, but rows already removed by a committed DELETE are gone. That's exactly why the gate stops those before apply.
